# Binary Outcome Matching Engine

This is a C++20 toy matching engine for a binary outcome market. It exposes a small REST API for submitting and cancelling limit orders, matching by price-time priority, supporting partial fills, and returning aggregated order book snapshots.

The implementation is intentionally in-memory. It prioritizes deterministic matching behavior, clear edge-case handling, and reviewable low-latency-oriented data structures over persistence or production exchange infrastructure.

## Features

- Submit BUY and SELL limit orders.
- Cancel active orders.
- Match crossed orders by price-time priority.
- Support full fills and partial fills.
- Return order status and fill details from submit/cancel calls.
- Return aggregated bid/ask snapshots.
- Reject duplicate order IDs.
- Reject invalid price, quantity, user, and side inputs.
- Prevent self-trades atomically by rejecting the incoming order before any fill occurs.
- Serialize REST access to the in-memory engine with a mutex to avoid cancel/snapshot races.
- Provide assertion-based API tests for the required edge cases.
- Provide a timestamped scenario emulator that logs real API inputs, responses, trades, and book states.

## Running The Application

```bash
cmake -S . -B build
cmake --build build
./build/matching_engine
```

The server always binds to port `8080`. If port `8080` is already in use, the process exits with a warning so you can stop the existing server and rerun it.

Keep the server terminal open while running tests from a second terminal.
## Testing And Demo

This project provides two main user-facing ways to exercise the API: a strict assertion suite and a readable scenario emulator.

### Option 1: Assertion Test Suite

Use this when you want a fast pass/fail correctness check:

```bash
python3 test_suite.py
```

Expected result:

```text
PASS test_resting_orders_and_snapshot
PASS test_full_fill_uses_resting_price
PASS test_partial_fill_accounting_and_cancel
PASS test_price_time_priority
PASS test_self_trade_rejects_incoming_atomically
PASS test_validation_and_terminal_order_cancels
PASS 6 test groups
```

The assertion suite resets the in-memory book between test groups and fails if any required behavior regresses. It covers:

- resting bid/ask snapshot behavior
- full fills
- resting-order trade price
- partial fill accounting
- cancellation of partially filled orders
- price priority
- FIFO time priority
- atomic self-trade rejection
- duplicate order IDs
- invalid side, price, and quantity
- unknown-order cancellation
- filled-order cancellation rejection

### Option 2: Readable Scenario Emulator

Use this when you want to see realistic order flow step by step:

```bash
python3 test_suite.py --demo
```

The emulator uses the same REST API as the assertion suite, but prints a human-readable walkthrough. It includes:

- scenario names
- nanosecond timestamps
- request method and path
- order payloads
- HTTP response status and elapsed request time
- generated trade details
- order status checks
- compact order book snapshots


## API

### Health

```http
GET /health
```

### Submit Order

```http
POST /order
Content-Type: application/json

{
  "order_id": 1,
  "user_id": 100,
  "side": "BUY",
  "price": 50,
  "qty": 10
}
```

Prices are integer cents from `1` to `99`. Quantity must be positive. The response includes status, remaining quantity, and trades generated by the incoming order.

### Cancel Order

```http
DELETE /order/1
```

Cancelling an open or partially filled order returns `200`. Unknown orders return `404`. Filled or already-cancelled orders return `409`.

### Get Order

```http
GET /order/1
```

Returns order details and status for accepted orders.

### Order Book Snapshot

```http
GET /orderbook
```

Returns aggregated price levels:

```json
{
  "bids": [{ "price": 50, "volume": 70 }],
  "asks": [{ "price": 60, "volume": 10 }]
}
```

### Reset

```http
POST /reset
```

This is a development/test endpoint used by `test_suite.py` to isolate test cases. It would not be exposed on a production public API.

## Internal Design

The core order book uses fixed price ladders:

- `PriceLevel bids_[100]`
- `PriceLevel asks_[100]`

Binary outcome prices are bounded to `1..99`, so array-backed price levels avoid tree or heap overhead. Best bid and best ask pointers allow direct access to the top of book, and any best-price scan is bounded by 99 ticks.

Each price level is a doubly linked FIFO queue. That gives strict time priority at a price level and O(1) removal once an order pointer is known.

Orders are stored in a preallocated vector-backed pool. Accepted order IDs are indexed in:

- `all_orders_` for permanent order lookup and duplicate rejection.
- `active_orders_` for active cancel lookup.

The REST layer guards all engine operations with one mutex. This is a deliberate correctness choice for a REST toy service: submit, cancel, order lookup, reset, and snapshots cannot interleave with a match. A true low-latency deployment would normally replace the REST path with an order-entry sequencer and run the matcher on a single dedicated event loop.

## Matching Rules

- BUY crosses when `buy_price >= best_ask`.
- SELL crosses when `sell_price <= best_bid`.
- Trade price is the resting order price.
- Fill quantity is `min(incoming.leaves_qty, resting.leaves_qty)`.
- Fully filled resting orders are removed from the book.
- Partially filled resting orders remain at their original time priority.
- If an incoming order has remaining quantity after matching, it rests on the book.

## Self-Trade Prevention

The engine rejects an incoming order if it would match any crossing resting order from the same `user_id`.

This is atomic: the engine checks the crossing path before matching. If a self-trade would occur, no partial fills are performed and the incoming order is not recorded. This avoids a case where an order partially fills against other users and then hits the submitter's own resting order.

### Optional Raw JSONL Demo Output

Use this if you want machine-readable logs:

```bash
python3 test_suite.py --demo --jsonl
```

Each line is a JSON event with `ts_ns`, event type, scenario/request/response metadata, payloads, response bodies, elapsed request time, trades, and snapshots.

All three commands require the server to already be running on `localhost:8080`.
